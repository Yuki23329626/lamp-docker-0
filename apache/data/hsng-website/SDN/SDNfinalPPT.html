子計畫一: 雲端資料中心網路技術

  • 子計畫一以軟體定義網路技術在雲端資料中心的實體網路進行網路控制與管理，研究雲端資料中心中以軟體定義網路為基礎的網路技術，提供實體網路的各種解決方案以及提供高效能之Hybrid SDN 網路介接架構及運作機制，著眼於「Hybrid SDN 虛擬化網路」中有關服務品質(Quality of Service)之研究議題進行探討。
  • 網路技術設計目標
    • 大型資料中心網路須滿足下列規格:
    • 可支援100,000台主機(PM)， 1,000,000個虛擬主機(VM)。
    • 使用者可以規劃自己的私有IP位址,因此不同使用者的VM允許指定重複的IP。
    • 高速故障復原能力。
    • 具備Plug & Play 功能，網路交換機、PM 等設備可以任意放置與搬動所在地點,系統仍能自動偵測並進行自動設定。
    • 交換機與PM可以實際需要進行安裝，不需全部安裝始可使用。
    • 支援VM migration。
    • 支援Multi-tenant 且各個Tenant 可獨立設定VLAN 。
    • 具備動態網路負載平衡路由能力 (dynamic LB routing)。
    • 具備自動網路拓樸偵測與PM, VM 位置偵測能力。
    • 網路具備Address resolution。
  • Case 1: Hybrid Switches
    • 基本構想
      • 以SDN 交換機及SDN controller使網路具備中央控管特性，並解決傳統交換機FIB限制與tenant ID不足的問題。
      • 網路中大部分設備仍使用乙太網路交換機，以節省網路的建置成本。
      • 利用資料中心為一受控系統特性，重新安排MAC攜帶各類訊息，並利用SDN交換機具修改IP與MAC能力，充分利用各IP與MAC位址欄位攜帶路由資訊並避免FIB不足。
      • 利用實體server中的軟體交換機(OVS)協助動態路由，並且達到tenant isolation之目的。

  • Case 2: All-SDN-Switch DC
    • Typical infrastructure of a data center
    • All-SDN-switch data center
    • SDN switch
      • No auto address learning
      • No spanning tree protocol
      • No flow entry initially
      • Support LLDP
    • Hybrid in-band/out-of-band control
      • OFC connects to all core switches (top layer) (out-of-band)
      • OFC connects to all other switches through core switches (in-band)
    • Objectives
      • Traffic Engineering
      • Auto topology discovery
      • Auto switch/OVS/host(NIC) addressing
      • Auto routing
      • Load balancing routing
      • Fast failure detection and recovery
      • MPTCP
    • Components
      • OpenFlow Controller1：Addressing、Routing
      • OpenFlow Controller2：Fault management、Database update、VM location tracking
      • OpenStack Manager：VM and tenant management
      • Database
        • Switch與MAC Address的對應表
        • Link流量紀錄表 (metering)
        • VM位置的對應表 (IP, MAC, VLAN, OSV)
      • Auto topology discovery
        • Ryu (OFC) + LLDP
          • Controller會透過Packet Out Message送給已知的switch與其port數相符的封包，並命令switch將他從所有port轉送出去，未與Controller做Handshake的switch因不知道如何處理此封包（Table-miss Flow Entry），因此將封包送回Controller處理，Controller會建立收送之switch間port連接的link。
      • Auto switch/OVS/host(NIC) addressing
        • Assumptions
          • 3-layer switch architecture + PM with OVS
          • OFC knows core switches (via out-of-band control)
          • VMs are managed by Openstack manager (such as VM creation, migration, etc)
        • Topology-aware addressing
          Multicast/Unicast
          (1 bits)          		 Core SW ID
          (7 bits)          		 Region SW ID
          (1 byte)          		 Edge SW ID
          (1 byte)          		 Edge Port ID (NIC)
          (1 byte)          		 Tenant ID
          (2 bytes)
          Objective
          • auto routing based on destination MAC address
          • Load balancing by selecting proper destination MAC address
        • Address assignment results (multiple addresses per host/switch)
      • Auto routing
        • Consider VM1 send ARP to VM2
          • OVS(VM1) redirects ARP to OFC
          • OFC query DB to get a list of MAC addresses for OVS(VM1) (so is OVS(VM2)
          • Select all pairs of (MAC(OVS(VM1)), MAC(OVS(VM2)) with longest common prefix (LCP)
          • Among them, select the one with the least load (for basic load balancing mechanism)
          • Set the routing result via adding flow entry to OVS(VM1) and OVS(VM2)
          • After that, routing is based on the flow entry set at each switch (will be described later).

            • Assume (src:1.1.1.2,dst:1.2.2.2) is selected.

        • Edge switch
          • Destination is under the same edge switch (downward) (16*48=768 flow entries)
            • Destination Ouptport
            [Core SW ID][Region SW ID][Edge SW ID][Edge Port ID] (dl_dst=dst_addr/32) [Edge Port ID]
          • Needs to send to regional or core switch (flow entries = no. of cores)
            • Destination Ouptport
            [Core SW ID](dl_dst=dst_addr/8) Port to the core
        • Region switch
          • Destination is under the same switch (downward) (4*48=192 flow entries)
            • Destination Ouptport
            [Core SW ID][Region SW ID][Edge SW ID] (dl_dst=dst_addr/24) [Edge SW ID]
          • Needs to send to core switches (flow entries = num of cores)
            • Destination Ouptport
            [Core SW ID](dl_dst=dst_addr/8) Port to the core
        • Core switch
          • Destination is under the core switch (downward) (48 flow entries)
            • Destination Ouptport
            [Core SW ID][Region SW ID] (dl_dst=dst_addr/16) [Region Port ID]
      • Fast failure detection and recovery
        • Fault Detection
          • Periodic polling based on multi-rooted tree topology (one poll for each subtree)

A Summary for Staged Refresh Timers for RSVP

Introduction

RSVP relies on periodic refresh between routers to maintain reservation status.

Terminology

  1. Sending and Receiving Nodes :
    1. A sending node is a router or host that generates RSVP messages.
    2. A receiving node is defined as the RSVP router or host that is one hop away from a sending node.
  2. Trigger and Refresh Message
    1. Trigger Messages are generated by an RSVP host or a router due to state changes.
    2. Refresh Messages contain replicated(copyed) state information generated by a router to maintain state.

RSVP Extension

  1. Outline of Operation
    1. RSVP trigger message(Sending Node -> Receiving Node) : Insert a new echo-request flag into the RSVP common header of the message .
    2. Retransmit Trigger Message if the message is teardown message until echo-reply is received : Interval retransmission is govened by staged refresh timer(starts at a small interval and increase exponentially [(1+D)i*Rf] ) until reach a threshold. => 1. refresh PATH & RESV , 2. Stop re-transmitting tear-down message.
    3. Acknowledges the arrival of the message (Echo-Reply) (Receiving Node -> Sending Node)
    4. Receive Echo-Reply : Scale back the refresh rate of the flow .
    5. Cost :
      1. A new bit is defined in the flag field of the RSVP common header.
      2. Four new message types are created for echo-reply.
  2. Time Parameters
    1. Fast Refresh Interval : Rf :
      1. Initial retransmission interval for trigger messages
      2. The sending node will "schedule" a retransmission after Rf seconds
      3. Rf can be small as the round trip time (RTT) between a sending and receiving node, if known : defalut in this paper : 3 seconds.
    2. Slow Refresh Interval : Rs :
      1. The sending node retransmit with this interval after it has determined that receiving nodes support the RSVP echo-reply.
      2. To reduce the number of unnecessary refreshes in a stable network, Rs can be set to a large value : default in this paper is 15 minutes.
    3. Fixed Refresh Interval : R :
      1. A node retransmits the trigger message with the interval (1+D)i*Rf until
        1. (1+D)i*Rf >= R
        2. an echo reply has been received
      2. If no reply has been received, the node continues to retransmit refreshes every R seconds
      3. Default in this paper is 30 seconds (the same as current RSVP Specification)
    4. Increment Value : D :
      1. D governs the speed with which the sender increases the refresh interval.
      2. The ratio of two successive refresh intervals is 1+D
      3. Default in this paper is 0.30 (the same as Slew.Max in current RSVP Specification)
  3. Staged Refresh :
    1. Ri = (1+D)i*Rf if no reply and i <= I
    2. Ri = R if no reply and i > I
    3. Ri = Rs after echo reply
    4. I = (log(R / Rf)) / log(1+D)
